# The JavaScript Engine — How Your Code Actually Runs

Great starting point. Understanding the JS engine is one of those foundational things that makes *everything else* click — closures, the event loop, hoisting, async behavior. Let's build this up from the ground.

---

## What Even Is a JS Engine?

At its core, a JavaScript engine is a program that reads your JavaScript source code (which is just text) and executes it. The most famous one is  **V8** , built by Google, which powers both Chrome and Node.js. Others include SpiderMonkey (Firefox) and JavaScriptCore (Safari).

Here's the key insight that surprises a lot of people:  **JavaScript is not purely interpreted** . Older engines did interpret it line by line, but modern engines are far more sophisticated. V8, for example, uses a combination of compilation strategies to make your code run fast. This is often called  **Just-In-Time (JIT) compilation** .

---

## The Journey of Your Code Through the Engine

Think of it like a factory assembly line. Your raw JavaScript text enters one end, and executable machine code comes out the other. Here's what happens at each station.

### Step 1 — Parsing

The engine first needs to *understand* the structure of your code. It does this in two sub-steps.

**Lexical Analysis (Tokenization):** The raw source text is broken into meaningful chunks called  *tokens* . This is actually very similar to what the `Tokenizer` class in your PDF does! For example, the line `const x = 5 + 3;` becomes a stream of tokens like `[KEYWORD: const] [IDENTIFIER: x] [OPERATOR: =] [NUMBER: 5] [OPERATOR: +] [NUMBER: 3] [SEMICOLON]`. At this point, the engine doesn't care about meaning — it's just identifying the vocabulary.

**Syntactic Analysis (Building the AST):** Those tokens are then fed into a  *parser* , which checks that they follow the grammar rules of JavaScript and arranges them into a tree structure called an  **Abstract Syntax Tree (AST)** . Each node in the tree represents a language construct. The `=` becomes an assignment node with `x` on the left and an addition node (containing `5` and `3`) on the right. If your syntax is broken, the engine throws a `SyntaxError` here — before a single line of your code runs.

You can actually visualize any code's AST at [astexplorer.net](https://astexplorer.net/) — it's a great way to build intuition for this.

### Step 2 — Compilation and Interpretation (The JIT Dance)

Here's where V8 gets clever. It doesn't make a binary choice between "compile everything" or "interpret everything". Instead, it uses a two-tier strategy.

**Ignition (the Interpreter):** The AST is first handed to V8's interpreter called Ignition. Ignition walks the AST and generates **bytecode** — a compact, lower-level representation that's faster to execute than re-reading the raw source, but not yet machine code. This bytecode starts running  *immediately* . The goal here is fast startup time. Your code begins executing quickly rather than waiting for full compilation.

**TurboFan (the Optimizing Compiler):** While Ignition is running your bytecode, V8 is also *profiling* it — watching which parts of your code run frequently. Functions that get called a lot are called  **"hot" functions** . TurboFan picks these up and compiles them into highly optimized machine code specific to your CPU. Next time that hot function runs, it uses the fast machine code path instead of the bytecode path.

This is the essence of JIT: compile the parts that matter most, *just in time* for when they're needed.

### Step 3 — Optimization and Deoptimization

TurboFan makes *assumptions* while optimizing. For example, if it notices that a function has always been called with numbers, it optimizes for that case specifically. But JavaScript is dynamically typed — what happens if you suddenly call that function with a string?

The engine  **deoptimizes** : it throws away the optimized machine code and falls back to the bytecode path. This is why type consistency in hot paths is genuinely important for performance, not just a code style preference.

## A Mental Model to Hold Onto

Imagine the engine like this:

```
Your Source Code (.js file)
         ↓
    [ Tokenizer ]  →  stream of tokens
         ↓
     [ Parser ]    →  AST (tree of meaning)
         ↓
   [ Ignition ]    →  bytecode (starts executing immediately)
         ↓ (hot paths detected)
   [ TurboFan ]   →  optimized machine code (fast path)
```

The crucial thing to appreciate is that by the time *your* code runs, the engine has already done a full pass over it during parsing. This is why things like **hoisting** happen — the engine knows about your `var` declarations and `function` declarations before execution begins, because it built the AST first.

---

## Why Does This Matter Practically?

Understanding the engine explains several JS behaviors that otherwise seem like magic:

**Hoisting** happens because the parser processes declarations before the execution phase begins. The engine *knows* `function foo() {}` exists before line 1 of your code runs.

**Hidden classes** in V8 mean that adding properties to objects in a consistent order makes them faster, because the engine can treat them like typed structs internally.

**The event loop** (which we should definitely cover next) exists because the engine itself is single-threaded — it has one call stack — and needs a mechanism to handle async work without freezing.
