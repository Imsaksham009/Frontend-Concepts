# Call Stack, Memory Heap, Stack Overflow, and Garbage Collection

Let's build this up carefully, because these concepts are deeply interconnected. By the end, you'll be able to *visualize* what's happening in memory as your code runs — and that changes how you write code.

---

## The Two Memory Regions

When the JS engine runs your program, it needs two fundamentally different kinds of storage. Think of your computer's memory as a giant warehouse, and the engine carves out two distinct sections of it for different purposes.

**The Memory Heap** is a large, unstructured region of memory used for dynamic allocation. It's where objects, arrays, functions, and closures live. It's called a "heap" because things get thrown in somewhat randomly — there's no strict ordering. When you write `const person = { name: 'Peter' }`, that object gets allocated somewhere in the heap, and the variable `person` holds a *reference* (a memory address) pointing to it.

**The Call Stack** is a small, tightly structured region used to track *execution context* — essentially, "where am I in the program right now, and what are my local variables?" It follows a strict Last-In-First-Out (LIFO) rule, like a stack of plates. You can only add to the top and remove from the top. Each time you call a function, a new "plate" (called a  **stack frame** ) is placed on top. When the function returns, that frame is removed.

These two regions serve completely different purposes: the heap stores  *data* , and the call stack tracks  *execution flow* .

---

## The Call Stack in Depth

Let's trace through a real example so you can see exactly what happens frame by frame.

```javascript
function greet(name) {
    const message = 'Hello, ' + name; // 'message' lives in this frame
    return shout(message);            // pushes a new frame on top
}

function shout(text) {
    return text.toUpperCase();        // executes, then this frame pops off
}

const result = greet('Peter');        // this is where everything starts
```

When execution begins, the engine creates a **Global Execution Context** and pushes it onto the stack as the base frame. This frame holds your global variables like `result`.

Then `greet('Peter')` is called. A new frame is pushed onto the stack for `greet`, containing its local variable `message` and its argument `name`. Crucially, these primitive values (`'Hello, Peter'`, `'Peter'`) are stored *directly inside the stack frame* because they are primitives.

Then inside `greet`, `shout(message)` is called. Another frame is pushed for `shout`, containing `text`. Now the stack looks like this, from bottom to top: `[Global] → [greet] → [shout]`.

`shout` finishes and returns the uppercased string. Its frame is **popped** off. Control returns to `greet`, which then returns the result. `greet`'s frame is popped. `result` in the global frame gets assigned the final value.

This is a key insight about the call stack:  **it is the engine's memory of "where to return to" after each function finishes** . Each frame knows exactly which line of the calling function to go back to.

### Primitives Live on the Stack, Objects Live in the Heap

This distinction is incredibly important and explains a lot of JavaScript's behavior.

```javascript
// Primitives — stored directly in the stack frame
let a = 10;
let b = a;   // b gets a COPY of the value 10
b = 20;
console.log(a); // still 10 — a and b are completely independent

// Objects — the stack holds a REFERENCE to the heap
let obj1 = { score: 100 };
let obj2 = obj1;    // obj2 gets a COPY of the *reference*, not the object
obj2.score = 999;
console.log(obj1.score); // 999 — both point to the SAME heap object!
```

When you assign `let b = a`, the value `10` is copied directly because it fits neatly in a stack frame. But when you assign `let obj2 = obj1`, you're copying the *memory address* (reference) stored in `obj1`. Both variables now point to the same object sitting in the heap. This is why mutating `obj2` also appears to mutate `obj1` — you're modifying the single object they both reference.

## Stack Overflow

Now that you understand how frames stack up, stack overflow becomes intuitive. The call stack has a fixed maximum size — it's a small, bounded region of memory by design (typically a few megabytes). If you keep pushing frames without ever popping them, you'll eventually run out of space.

The most common cause is unbounded recursion — a function that calls itself without a proper base case:

```javascript
function infinite() {
    // No base case — this will call itself forever
    return infinite();
}

infinite(); // RangeError: Maximum call stack size exceeded
```

Every call to `infinite()` pushes a new frame. Since `infinite` never returns, frames are never popped. Eventually the stack is completely full and the engine throws a `RangeError`. This is a stack overflow — it's not about running out of RAM; it's specifically about exhausting the call stack's capacity.

A subtler and more realistic example is deep recursion on large data. Consider processing a deeply nested object recursively — if nesting goes thousands of levels deep, you'll hit the same limit even if your logic is otherwise correct. This is why some of the implementations in your PDF (like Deep Flatten) offer both recursive and iterative approaches. The iterative version uses a manually managed array as a stack on the  *heap* , which doesn't have the same size constraint.

## The Memory Heap and Garbage Collection

Here's a question worth sitting with for a moment: if objects live in the heap, and the heap is just a region of memory, how does that memory ever get freed? You never explicitly call `free(obj)` in JavaScript like you would in C. Something has to clean up after you.

That something is the  **Garbage Collector (GC)** . It runs automatically, periodically scanning the heap to find objects that your program can no longer possibly use, and reclaiming that memory.

The central concept the GC uses is  **reachability** . An object is considered "alive" (and kept in memory) if it is reachable — meaning there is some chain of references from a *root* to that object. Roots are things like global variables, the current call stack (local variables in active frames), and closures. If you can trace a path from any root to an object, it stays. If no such path exists, the GC considers it garbage.

### How Modern GCs Work: Mark and Sweep

The dominant algorithm used in V8 is called  **Mark and Sweep** , and it runs in two phases.

In the  **Mark phase** , the GC starts from all roots and traverses every reference — like following every link in a graph. Every object it can reach gets "marked" as alive.

In the  **Sweep phase** , the GC scans the entire heap. Any object that was *not* marked gets swept away — its memory is reclaimed and returned to the pool for future allocations.

V8 also uses a **generational** approach, dividing the heap into a "young generation" (for newly created objects) and an "old generation" (for objects that have survived multiple GC cycles). Most objects die young — you create a temporary object inside a function, the function returns, and the object immediately becomes unreachable. The GC exploits this pattern by scanning the young generation very frequently and cheaply, while scanning the old generation less often but more thoroughly.

## Memory Leaks — When the GC Can't Help You

The GC is excellent at cleaning up objects that are genuinely unreachable. But a memory leak in JavaScript doesn't mean you forgot to free something — it means you *accidentally kept a reference* to something you no longer need. The GC sees a live reference and dutifully keeps the object alive, even though from your program's logical perspective, you're done with it.

Let me show you the most common patterns.

**Accidental Global Variables** are perhaps the most straightforward. In non-strict mode, assigning to an undeclared variable creates a global. Globals are always reachable (they're roots), so they never get collected.

```javascript
function processData() {
    // Missing 'let' or 'const' — this creates a global variable!
    // It will live for the entire lifetime of the page.
    leakedData = new Array(1000000).fill('oops');
}
```

**Forgotten Event Listeners** are one of the most common real-world leaks, especially in single-page applications. When you attach an event listener to a DOM element, the callback function forms a closure over its surrounding scope. That closure keeps everything it references alive.

```javascript
function setupComponent() {
    const heavyObject = fetchMassiveDataset(); // large object in heap

    document.getElementById('btn').addEventListener('click', () => {
        // This closure captures 'heavyObject'
        // Even if you remove the button from the DOM visually,
        // the listener (and therefore heavyObject) stays in memory
        // unless you explicitly call removeEventListener
        console.log(heavyObject);
    });
}
```

**Closures holding stale references** are a subtler trap. Closures are a feature, not a bug, but they can accidentally extend the lifetime of objects far beyond what you intended.

```javascript
function createCache() {
    const cache = new Map(); // this Map lives on the heap

    return {
        add(key, value) { cache.set(key, value); },
        get(key) { return cache.get(key); }
        // No 'clear' method — the cache grows forever
        // and the closure keeps it alive as long as the
        // returned object is alive
    };
}

const myCache = createCache(); // myCache holds a reference to the closure
// which holds a reference to 'cache'
// If you keep adding to myCache without ever clearing it,
// that Map grows indefinitely in the heap
```

**Detached DOM nodes** are a classic React/DOM leak. You remove a node from the DOM, but if a JavaScript variable still holds a reference to it, it stays in memory along with everything it references.

```javascript
let detachedList;

function createList() {
    detachedList = document.createElement('ul'); // stored in a global
    for (let i = 0; i < 10000; i++) {
        detachedList.appendChild(document.createElement('li'));
    }
    document.body.appendChild(detachedList);
}

function removeList() {
    document.body.removeChild(detachedList);
    // The list is gone from the visible DOM
    // But 'detachedList' still holds the reference!
    // The entire list and all 10,000 li elements stay in the heap
  
    // Fix: also set detachedList = null;
}
```

### How to Think About Preventing Leaks

A useful mental model is to ask yourself: *"Can the garbage collector trace a path from a root to this object?"* If yes, it lives. So the question becomes: *"Are any of those paths ones I didn't intend to keep?"*

The fixes usually come down to a few patterns: explicitly setting references to `null` when you're done (`detachedList = null`), calling `removeEventListener` before removing components, using `WeakMap` or `WeakSet` for caches (these hold *weak* references that don't prevent GC), and being deliberate about what closures capture.

---

## Putting It All Together

Here's the complete picture of how these pieces interact during a function call:

When you call a function, the engine pushes a **stack frame** onto the call stack, containing the function's local primitives and *references* to any objects. Those objects themselves live in the  **heap** . When the function returns, the frame is popped — the primitives disappear immediately. The objects in the heap only disappear once the GC determines that no reachable reference points to them anymore. If a closure or global variable captured a reference to those objects, they stay alive indefinitely — which is either a feature (closures) or a bug (memory leak), depending on your intent.

Understanding this chain — how the stack and heap work together, and how the GC decides what to keep — gives you a fundamentally clearer model of what JavaScript is doing with your data at runtime.
