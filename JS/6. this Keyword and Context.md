# `this` Keyword and Context — The Complete Guide

---

## What Is `this` Fundamentally?

`this` is a special keyword in JavaScript that is  **automatically defined in every execution context** . It is not a variable you declare. It is not fixed at write time. It is a **dynamic reference** that the JavaScript engine sets automatically based on one thing and one thing only:

**How the function is called — not where it is written, not what it is named, not where it is defined.**

This is the single sentence you need to tattoo in your brain. Every confusion about `this` comes from forgetting this rule.

Think of `this` as an implicit argument that gets passed to every function automatically. When you call a method on an object, the object gets passed in as `this`. When you call a standalone function, something else gets passed in as `this` (depending on strict mode). `this` is just the engine's way of telling the function  *"here is the context you are operating in"* .

---

## The Four Rules That Determine `this`

There are exactly four rules that determine what `this` is. They have a priority order — when multiple rules could apply, the higher priority rule wins. Learn these four rules and their priority, and you can determine `this` in any situation without guessing.

**Priority order (highest to lowest):**

1. `new` binding
2. Explicit binding (`call`, `apply`, `bind`)
3. Implicit binding (method call)
4. Default binding (standalone function call)

Let's go through each one thoroughly.

---

### Rule 4 — Default Binding (Lowest Priority)

This applies when a function is called as a plain standalone function — not as a method, not with `new`, not with `call`/`apply`/`bind`.

```javascript
function showThis() {
    console.log(this);
}

showThis(); // What is 'this' here?
```

In  **non-strict mode** : `this` is the  **global object** . In browsers that's `window`. In Node.js that's `global`.

In  **strict mode** : `this` is  **`undefined`** .

```javascript
// Non-strict mode
function nonStrict() {
    console.log(this); // window (browser) or global (Node)
}
nonStrict();

// Strict mode
function strict() {
    'use strict';
    console.log(this); // undefined
}
strict();
```

The strict mode behavior exists for a good reason — defaulting to global was a major source of accidental global variable pollution. When `this` is `undefined` in strict mode, trying to use it throws a `TypeError` immediately, catching bugs at the source.

---

### Rule 3 — Implicit Binding

This applies when a function is called as a method of an object — using dot notation or bracket notation.

```javascript
const peter = {
    name: 'Peter Parker',
    greet() {
        console.log(this.name); // 'this' is the object before the dot
    }
};

peter.greet(); // 'Peter Parker' — 'this' is 'peter'
```

The rule is simple:  **the object before the dot at call time becomes `this`** .

Note "at call time" — this matters enormously. The function definition doesn't matter. Only the call site matters.

```javascript
function greet() {
    console.log(this.name);
}

const peter = { name: 'Peter Parker', greet };
const tony  = { name: 'Tony Stark',   greet };

// Same function — completely different 'this'
peter.greet(); // 'Peter Parker'
tony.greet();  // 'Tony Stark'
```

#### Implicit Binding Loss — The Most Common `this` Bug

This is where most developers get burned. When you extract a method from an object and call it separately, you **lose** the implicit binding:

```javascript
const peter = {
    name: 'Peter Parker',
    greet() {
        console.log(this.name);
    }
};

peter.greet(); // 'Peter Parker' — implicit binding, works fine

// Extract the method
const extractedGreet = peter.greet;

// Now call it standalone — default binding applies!
extractedGreet(); // undefined (strict) or '' (non-strict, window.name)
```

When you do `const extractedGreet = peter.greet`, you're just copying a reference to the function. The function itself has no memory of being part of `peter`. When called standalone, it follows default binding rules.

This exact bug appears in these common real-world situations:

```javascript
const counter = {
    count: 0,
    increment() {
        this.count++;
        console.log(this.count);
    }
};

// BUG 1 — Passing method as callback loses 'this'
setTimeout(counter.increment, 1000);
// TypeError or NaN — 'this' inside setTimeout callback
// is window/undefined, not counter

// BUG 2 — Event listener loses 'this'
button.addEventListener('click', counter.increment);
// 'this' inside handler is the button element, not counter

// BUG 3 — Array method callback loses 'this'
[1, 2, 3].forEach(counter.increment);
// 'this' is undefined/window, not counter
```

All three of these are the same problem — the method is being passed as a value and called without its object context.

---

### Rule 2 — Explicit Binding

This applies when you use `call`, `apply`, or `bind` to directly specify what `this` should be. This overrides implicit binding.

#### `call` — Invoke immediately, pass args individually

```javascript
function introduce(role, city) {
    console.log(`${this.name} is ${role} from ${city}`);
}

const peter = { name: 'Peter Parker' };
const tony  = { name: 'Tony Stark' };

introduce.call(peter, 'Spiderman', 'New York');
// 'Peter Parker is Spiderman from New York'

introduce.call(tony, 'Iron Man', 'Malibu');
// 'Tony Stark is Iron Man from Malibu'
```

#### `apply` — Invoke immediately, pass args as array

```javascript
introduce.apply(peter, ['Spiderman', 'New York']);
// 'Peter Parker is Spiderman from New York'

// Practical use — spreading an array into a function
const numbers = [5, 2, 8, 1, 9, 3];
Math.max.apply(null, numbers); // 9
// Modern equivalent: Math.max(...numbers)
```

#### `bind` — Returns new function with `this` permanently locked

`bind` does not call the function. It returns a **new function** where `this` is permanently set and cannot be overridden:

```javascript
function introduce(role, city) {
    console.log(`${this.name} is ${role} from ${city}`);
}

const peter = { name: 'Peter Parker' };

// Create a new function with 'this' permanently set to peter
const peterIntroduce = introduce.bind(peter);

peterIntroduce('Spiderman', 'New York');
// 'Peter Parker is Spiderman from New York'

// Even if you try to call it on another object — 'this' stays as peter
const tony = { name: 'Tony Stark', introduce: peterIntroduce };
tony.introduce('Iron Man', 'Malibu');
// STILL 'Peter Parker is Iron Man from Malibu'
// bind wins over implicit binding
```

`bind` also supports **partial application** — pre-filling some arguments:

```javascript
function multiply(a, b) {
    return a * b;
}

const double = multiply.bind(null, 2); // 'this' is null (irrelevant here), 'a' is always 2
double(5);  // 10
double(10); // 20
double(7);  // 14
```

#### Fixing the Implicit Binding Loss with `bind`

```javascript
const counter = {
    count: 0,
    increment() {
        this.count++;
        console.log(this.count);
    }
};

// Fix all three bugs from before using bind
setTimeout(counter.increment.bind(counter), 1000); // works — 1
button.addEventListener('click', counter.increment.bind(counter)); // works
[1, 2, 3].forEach(counter.increment.bind(counter)); // works
```

### Rule 1 — `new` Binding (Highest Priority)

When you call a function with the `new` keyword, four things happen automatically inside the engine:

1. A brand new empty object is created
2. That new object is set as `this` for the function
3. The new object's prototype is linked to the constructor's `prototype` property
4. The new object is returned automatically (unless you explicitly return a different object)

```javascript
function Person(name, age) {
    // 'this' is the brand new empty object
    this.name = name; // properties added to the new object
    this.age = age;
    // new object implicitly returned
}

const peter = new Person('Peter Parker', 25);
// peter = { name: 'Peter Parker', age: 25 }

console.log(peter.name); // 'Peter Parker'
```

`new` binding has the highest priority — it overrides all other rules:

```javascript
function Person(name) {
    this.name = name;
    console.log(this);
}

const obj = { name: 'Existing Object' };

// Even though we bind to obj, 'new' overrides it
const BoundPerson = Person.bind(obj);
const result = new BoundPerson('Peter');
// 'this' is the NEW object, not obj
// result = { name: 'Peter' }
```

## The Priority in Action — Determining `this` at Any Call Site

Here's a decision tree you can apply to any function call to determine `this`:

```
Is the function called with 'new'?
    YES → 'this' is the newly created object
    NO  ↓
Is the function called with call/apply/bind?
    YES → 'this' is whatever you explicitly passed
    NO  ↓
Is the function called as a method (obj.fn())?
    YES → 'this' is the object before the dot
    NO  ↓
Default: strict mode? 'this' = undefined
         non-strict?  'this' = global object
```

## Arrow Functions and `this` — Lexical Context

Arrow functions completely opt out of the four rules above. They  **never have their own `this`** . Instead, they capture `this` from the **lexical scope** where they are defined — just like a regular variable via the scope chain.

```javascript
const obj = {
    name: 'Peter',
  
    regularMethod() {
        console.log(this.name); // 'Peter' — implicit binding
      
        // Regular function callback — loses 'this'
        setTimeout(function() {
            console.log(this.name); // undefined — default binding
        }, 100);
      
        // Arrow function callback — captures 'this' from regularMethod's scope
        setTimeout(() => {
            console.log(this.name); // 'Peter' — lexical 'this'
        }, 100);
    }
};

obj.regularMethod();
```

The arrow function's `this` is determined when it is  **created** , not when it is  **called** . It locks onto whatever `this` was in the surrounding scope at the moment of creation.

This is why arrow functions are perfect for callbacks — they don't have their own `this` to mess up the context:

```javascript
class Timer {
    constructor() {
        this.seconds = 0;
    }
  
    start() {
        // Arrow function captures 'this' from start()'s context
        // which is the Timer instance
        setInterval(() => {
            this.seconds++; // correctly refers to Timer instance
            console.log(this.seconds);
        }, 1000);
    }
}

const timer = new Timer();
timer.start(); // 1, 2, 3, 4...
```

### When NOT to Use Arrow Functions for Methods

Because arrow functions capture `this` from their definition scope, they're wrong for object methods:

```javascript
const obj = {
    name: 'Peter',
  
    // WRONG — arrow function defined in global/module scope
    // 'this' captured from global scope, not obj
    greet: () => {
        console.log(this.name); // undefined — 'this' is global
    },
  
    // RIGHT — regular function, 'this' determined by call site
    greetCorrect() {
        console.log(this.name); // 'Peter'
    }
};

obj.greet();        // undefined
obj.greetCorrect(); // 'Peter'
```

## `this` in Classes

Classes use strict mode automatically, and `this` follows the same rules — but there are some important nuances:

```javascript
class Person {
    constructor(name) {
        this.name = name; // 'this' is the new instance
    }
  
    greet() {
        console.log(`Hello, I'm ${this.name}`);
    }
  
    greetArrow = () => {
        // Class field arrow function — 'this' is permanently
        // bound to the instance at construction time
        console.log(`Hello, I'm ${this.name}`);
    }
}

const peter = new Person('Peter');
peter.greet();       // "Hello, I'm Peter"
peter.greetArrow();  // "Hello, I'm Peter"

// Extracting and calling standalone
const greet = peter.greet;
greet(); // TypeError — 'this' is undefined in strict mode

const greetArrow = peter.greetArrow;
greetArrow(); // "Hello, I'm Peter" — arrow keeps 'this' as peter
```

The class field arrow function pattern (`greetArrow = () => {}`) is the modern, clean solution to the method extraction problem. Each instance gets its own copy of the function with `this` permanently bound to that instance. This is the pattern React class components used extensively:

```javascript
class Button extends React.Component {
    // Without this pattern, 'this' would be lost when React calls handleClick
    handleClick = () => {
        this.setState({ clicked: true }); // 'this' is always the component instance
    }
  
    render() {
        return <button onClick={this.handleClick}>Click</button>;
    }
}
```

## `this` in Special Situations

### In Event Handlers

```javascript
const button = document.querySelector('button');

// Regular function — 'this' is the element that received the event
button.addEventListener('click', function() {
    console.log(this); // <button> element
    this.style.color = 'red'; // works fine
});

// Arrow function — 'this' is captured from surrounding scope
button.addEventListener('click', () => {
    console.log(this); // window/undefined — NOT the button
    this.style.color = 'red'; // TypeError
});
```

For event handlers where you need to reference the element, use regular functions.

### In `setTimeout` and `setInterval`

```javascript
// Non-strict, regular function — 'this' is window
setTimeout(function() {
    console.log(this); // window
}, 100);

// Arrow function — captures 'this' from where setTimeout was called
const obj = {
    name: 'Peter',
    start() {
        setTimeout(() => {
            console.log(this.name); // 'Peter' — captured from start()
        }, 100);
    }
};
```

### Chained Methods — `this` at Each Step

```javascript
const calculator = {
    value: 0,
  
    add(n) {
        this.value += n;
        return this; // return 'this' to enable chaining
    },
  
    multiply(n) {
        this.value *= n;
        return this;
    },
  
    subtract(n) {
        this.value -= n;
        return this;
    },
  
    result() {
        return this.value;
    }
};

calculator
    .add(10)       // this = calculator, value = 10
    .multiply(3)   // this = calculator, value = 30
    .subtract(5)   // this = calculator, value = 25
    .result();     // 25
```

Returning `this` from each method is what enables method chaining — it's how jQuery, Lodash chains, and many builder patterns work.

## The Complete Mental Model

Here is the complete picture to hold in your head:

`this` is not about the function itself. It's not about where the function is written. It's purely about the **call site** — the moment and manner of invocation.

Every time you see a function call, ask: *"How is this function being called right now?"* Then apply the four rules in priority order.

The one exception is arrow functions — for those, ask: *"Where was this arrow function written, and what was `this` in that surrounding scope at that moment?"* Arrow functions freeze `this` from their birth environment and carry it forever, immune to all four rules.

This mental shift — from thinking of `this` as something the function owns, to thinking of it as something the call site provides — is what makes `this` go from confusing to completely predictable.
