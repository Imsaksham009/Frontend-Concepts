## Hoisting — What's Really Happening

Before any of your code executes, remember from our engine discussion — the parser does a full pass over your code and builds the AST. During this parsing phase, the engine also does something called **hoisting**. But hoisting is widely misunderstood, so let's be precise about what it actually means.

**Hoisting is not about code moving anywhere.** Your code doesn't physically relocate. What actually happens is that during the **creation phase** of an execution context, before a single line runs, the engine scans for all declarations and sets them up in memory first. Then in the **execution phase**, it runs your code top to bottom.

Every execution context (global or function) has two phases:

**Creation Phase:**

- Scans for all variable and function declarations
- Allocates memory for them
- Sets initial values (which varies depending on `var`, `let`, `const`, or `function`)

**Execution Phase:**

- Runs code line by line
- Assigns actual values

This two-phase model is the real mechanism behind hoisting. Let's see how it behaves differently for different declaration types.

---

### `var` Hoisting — Declared and Initialized to `undefined`

```javascript
console.log(name); // undefined — NOT a ReferenceError
var name = 'Peter';
console.log(name); // 'Peter'
```

During the creation phase, the engine sees `var name` and allocates memory for it, immediately initializing it to `undefined`. So when execution reaches `console.log(name)`, `name` exists — it's just `undefined`. The *assignment* (`= 'Peter'`) happens in the execution phase, at the line where it's written.

This is why `var` hoisting is considered dangerous — you can access a variable before its assignment without getting an error, leading to subtle bugs that are hard to trace.

---

### `let` and `const` Hoisting — The Temporal Dead Zone

```javascript
console.log(age); // ReferenceError: Cannot access 'age' before initialization
let age = 25;
```

Here's the nuance most people miss: **`let` and `const` ARE hoisted** — the engine does allocate memory for them during the creation phase. But unlike `var`, they are **not initialized**. They sit in what's called the **Temporal Dead Zone (TDZ)** — a state where the binding exists but has no value and cannot be accessed. Any attempt to read or write them before their declaration line throws a `ReferenceError`.

The TDZ exists from the start of the block scope until the declaration is reached during execution. This is intentional — it prevents the confusing `undefined` behavior of `var` and forces you to declare before use.

```javascript
{
    // TDZ for 'score' begins here
    console.log(score); // ReferenceError — in TDZ
    let score = 100;    // TDZ ends here, score is now initialized
    console.log(score); // 100
}
```

---

### Function Declaration Hoisting — Fully Hoisted

```javascript
// This works! Function declarations are fully hoisted
greet('Peter'); // 'Hello Peter'

function greet(name) {
    console.log('Hello ' + name);
}
```

Function declarations are the most completely hoisted of all. During the creation phase, the engine not only allocates memory for the function — it stores the **entire function definition**. The name AND the body are available from the very start of the execution context. This is why you can call a function declaration before it appears in your code.

---

### Function Expression Hoisting — Only the Variable is Hoisted

```javascript
// This FAILS
greet('Peter'); // TypeError: greet is not a function

var greet = function(name) {
    console.log('Hello ' + name);
}
```

A function expression assigns a function to a variable. The `var greet` part is hoisted (and initialized to `undefined`), but the function itself is not. So when you try to call `greet('Peter')` before the assignment, you're essentially calling `undefined('Peter')` — hence `TypeError: greet is not a function`.

With `let` or `const`, you'd get a `ReferenceError` instead, because the variable is in the TDZ.
