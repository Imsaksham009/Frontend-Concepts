## How Functions Are Invoked — The Four Patterns

How you *call* a function determines what `this` refers to inside it. This is one of the most important things to understand about JavaScript functions.

### Pattern 1 — Regular Function Invocation

```javascript
function showThis() {
    console.log(this);
}

showThis(); 
// In non-strict mode: window (browser) or global (Node)
// In strict mode: undefined
```

When a function is called without any context object, `this` defaults to the global object in non-strict mode, or `undefined` in strict mode. This is often the source of bugs — methods accidentally called without their object context lose their `this`.

### Pattern 2 — Method Invocation

```javascript
const spiderman = {
    name: 'Peter Parker',
    greet() {
        console.log(this.name); // 'this' is the spiderman object
    }
}

spiderman.greet(); // 'Peter Parker'

// But watch what happens when you extract the method
const fn = spiderman.greet;
fn(); // undefined — 'this' is now global/undefined!
```

When a function is called as a method of an object (with the dot notation), `this` is set to the object before the dot. But the moment you detach the method from the object and call it standalone, `this` is lost. This is the classic `this` confusion problem.

### Pattern 3 — Constructor Invocation with `new`

```javascript
function Person(name, age) {
    // When called with 'new', 'this' is a brand new empty object
    this.name = name;
    this.age = age;
    // The new object is implicitly returned
}

const peter = new Person('Peter', 25);
console.log(peter.name); // 'Peter'
```

When you use the `new` keyword, four things happen automatically:

1. A brand new empty object is created
2. `this` inside the function is set to that new object
3. The new object's prototype is linked to the constructor's `prototype` property
4. The new object is implicitly returned (unless you explicitly return a different object)

### Pattern 4 — Explicit Invocation with `call`, `apply`, `bind`

These are the methods from your PDF. They let you explicitly set what `this` should be.

```javascript
function introduce(role, city) {
    return `${this.name} - ${role} - ${city}`;
}

const peter = { name: 'Peter Parker' };

// call — arguments passed individually
introduce.call(peter, 'Spiderman', 'NYC');
// 'Peter Parker - Spiderman - NYC'

// apply — arguments passed as array
introduce.apply(peter, ['Spiderman', 'NYC']);
// 'Peter Parker - Spiderman - NYC'

// bind — returns a NEW function with 'this' permanently set
const peterIntro = introduce.bind(peter, 'Spiderman');
peterIntro('NYC'); // 'Peter Parker - Spiderman - NYC'
peterIntro('LA');  // 'Peter Parker - Spiderman - LA'
```

The key difference with `bind` is that it doesn't call the function — it returns a new function with `this` locked in. This is incredibly useful for callbacks, where you need to preserve context.

## Arguments — Everything You Need to Know

### The `arguments` Object

In regular (non-arrow) functions, there's a special object called `arguments` available inside every function call — an array-like object containing all the arguments passed, regardless of what parameters you declared.

```javascript
function sum() {
    console.log(arguments);
    // { 0: 1, 1: 2, 2: 3, length: 3 } — array-like, NOT an array
  
    let total = 0;
    for (let i = 0; i < arguments.length; i++) {
        total += arguments[i];
    }
    return total;
}

sum(1, 2, 3); // 6
sum(1, 2, 3, 4, 5); // 15 — works with any number of arguments
```

The `arguments` object is **not a real array** — it doesn't have array methods like `.map()` or `.filter()`. To convert it: `Array.from(arguments)` or `[...arguments]`.

`arguments` is largely superseded by rest parameters in modern JS, but you'll encounter it in legacy code.

### Rest Parameters — The Modern Way

```javascript
function sum(...numbers) {
    // 'numbers' is a REAL array
    return numbers.reduce((total, n) => total + n, 0);
}

sum(1, 2, 3, 4, 5); // 15

// You can combine named params with rest
function logInfo(level, ...messages) {
    messages.forEach(msg => console.log(`[${level}] ${msg}`));
}

logInfo('ERROR', 'File not found', 'Connection refused', 'Timeout');
// [ERROR] File not found
// [ERROR] Connection refused
// [ERROR] Timeout
```

Rest parameters collect all *remaining* arguments into a real array. Unlike `arguments`, they work in arrow functions, support all array methods, and make your intent explicit in the function signature.

### Default Parameters

```javascript
function createUser(name, role = 'viewer', active = true) {
    return { name, role, active };
}

createUser('Peter');
// { name: 'Peter', role: 'viewer', active: true }

createUser('Tony', 'admin');
// { name: 'Tony', role: 'admin', active: true }

// Default params can reference earlier params
function greet(name, greeting = `Hello, ${name}!`) {
    return greeting;
}

greet('Peter'); // 'Hello, Peter!'
```

Default parameters are evaluated at call time, not at definition time. This matters for objects and arrays as defaults — each call gets a fresh copy.

## Modern Function Forms — All of Them

### Function Declarations

```javascript
function add(a, b) {
    return a + b;
}
```

Fully hoisted. Has its own `this`. Has `arguments` object. Can be used as a constructor with `new`.

### Function Expressions

```javascript
const add = function(a, b) {
    return a + b;
};

// Named function expression — useful for recursion and debugging
const factorial = function fact(n) {
    return n <= 1 ? 1 : n * fact(n - 1); // can reference itself by name
};
```

Not hoisted (the variable is, but not the function). The name in a named function expression is only available *inside* the function itself.

### Arrow Functions — and How They Differ

Arrow functions are not just syntactic shorthand. They have fundamentally different behavior in several important ways.

```javascript
// Concise forms
const double = n => n * 2;
const add = (a, b) => a + b;
const getObj = () => ({ name: 'Peter' }); // wrap object in parens

// Block body form (needs explicit return)
const multiply = (a, b) => {
    const result = a * b;
    return result;
};
```

**The critical difference — `this` binding:**

Arrow functions do not have their own `this`. Instead, they **lexically capture** `this` from the surrounding scope where they were  *defined* . This is called lexical `this`.

```javascript
const timer = {
    count: 0,
  
    // Regular function — 'this' is lost when used as callback
    startBroken() {
        setInterval(function() {
            this.count++; // 'this' is window/undefined here!
            console.log(this.count); // NaN
        }, 1000);
    },
  
    // Arrow function — captures 'this' from startFixed's scope
    startFixed() {
        setInterval(() => {
            this.count++; // 'this' correctly refers to 'timer'
            console.log(this.count); // 1, 2, 3...
        }, 1000);
    }
}
```

Before arrow functions, the common pattern was `const self = this` or `.bind(this)`. Arrow functions made both unnecessary.

**Other arrow function differences:**

* No `arguments` object (use rest params instead)
* Cannot be used as constructors (calling with `new` throws TypeError)
* No `prototype` property
* Cannot be used as generator functions

### Immediately Invoked Function Expressions (IIFE)

```javascript
// Runs immediately, creates its own scope
(function() {
    const private = 'nobody can access this from outside';
    console.log('runs immediately');
})();

// Arrow function IIFE
(() => {
    console.log('also runs immediately');
})();

// Useful for async IIFEs — common pattern for top-level await
(async () => {
    const data = await fetchSomeData();
    console.log(data);
})();
```

IIFEs were critical before ES modules for creating private scopes and avoiding global variable pollution. You still see them frequently in bundled code.

### Generator Functions

```javascript
function* counter() {
    let i = 0;
    while (true) {
        yield i++;  // pauses here and returns i, resumes on next()
    }
}

const gen = counter();
gen.next(); // { value: 0, done: false }
gen.next(); // { value: 1, done: false }
gen.next(); // { value: 2, done: false }
// This generator is infinite but lazy — only computes when asked
```

Generator functions can pause and resume execution at `yield` points. They return an iterator object. The `yield` keyword is like a "pauseable return" — it sends a value out and waits until `.next()` is called again to resume.

### Async Functions

```javascript
async function fetchUser(id) {
    try {
        const response = await fetch(`/api/users/${id}`);
        const user = await response.json();
        return user; // this is automatically wrapped in a Promise
    } catch (error) {
        console.error('Failed:', error);
    }
}
```

An `async` function always returns a Promise. The `await` keyword pauses execution *of that function* (not the entire thread — remember the event loop) and waits for a Promise to settle, then resumes. Under the hood, `async/await` is syntactic sugar over Promises and the microtask queue — `await` is essentially `.then()` written in synchronous-looking style.

## First-Class Functions — Why This All Matters

JavaScript treats functions as  **first-class citizens** , meaning functions are just values. They can be:

```javascript
// Stored in variables
const greet = function(name) { return `Hello ${name}`; };

// Stored in objects
const utils = {
    double: n => n * 2,
    triple: n => n * 3
};

// Stored in arrays
const operations = [n => n + 1, n => n * 2, n => n ** 2];

// Passed as arguments — these are called "callbacks"
[1, 2, 3].map(n => n * 2); // [2, 4, 6]

// Returned from other functions — these create "closures"
function multiplier(factor) {
    return n => n * factor; // returns a function
}

const triple = multiplier(3);
triple(5); // 15
triple(10); // 30
```

This last pattern — functions returning functions — is the foundation of closures, currying (which is in your PDF), and higher-order functions like `debounce` and `throttle` (also in your PDF). When `multiplier` returns the inner arrow function, that inner function **closes over** `factor` — it remembers the value of `factor` from when it was created, even after `multiplier` has returned and its stack frame is gone.

This is the bridge between functions and closures, which is probably the next deep topic worth exploring. Want to dive into **closures, scope, and the scope chain** next? Everything in your PDF — debounce, throttle, currying, memoization — relies on a deep understanding of closures.
