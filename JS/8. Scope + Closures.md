# Closures — The Complete, Definitive Guide

Closures are arguably the most important concept in JavaScript. Nearly everything powerful in the language — callbacks, currying, memoization, debounce, throttle, module patterns, React hooks — is built on closures. Let's go so deep that you never need to look this up again.

---

## Start From the Ground Up — Scope

Before you can truly understand closures, you need to understand **scope** at a deep level. Scope answers one question: *"Where in your code can a particular variable be accessed?"*

JavaScript uses **Lexical Scoping** (also called Static Scoping). This means the scope of a variable is determined by  **where it is written in the source code** , not by where or how the function is called. The word "lexical" literally means "relating to the text" — the structure of your written code defines the scope rules, not the runtime behavior.

This is the foundation everything else is built on.

### The Three Levels of Scope

**Global Scope** — Variables declared outside any function or block. Accessible from everywhere in your program. They live for the entire lifetime of the program.

```javascript
const globalVar = 'I am everywhere';

function someFunction() {
    console.log(globalVar); // accessible — no problem
}

someFunction();
console.log(globalVar); // accessible here too
```

**Function Scope** — Variables declared inside a function. Only accessible within that function and any functions nested inside it. They are created when the function is called and destroyed when the function returns.

```javascript
function outer() {
    const functionScoped = 'only inside outer';
    console.log(functionScoped); // works fine
}

outer();
console.log(functionScoped); // ReferenceError — doesn't exist out here
```

**Block Scope** — Introduced with ES6. Variables declared with `let` and `const` inside any block `{}` (if statements, for loops, while loops, standalone blocks) are only accessible within that block.

```javascript
if (true) {
    var varVariable = 'I leak out';      // function scoped, not block scoped
    let letVariable = 'I stay inside';   // block scoped
    const constVariable = 'me too';      // block scoped
}

console.log(varVariable);   // 'I leak out' — var ignores block scope!
console.log(letVariable);   // ReferenceError
console.log(constVariable); // ReferenceError
```

This `var` leaking behavior is a classic source of bugs, especially inside loops.

## The Scope Chain

When you reference a variable inside a function, the engine doesn't just look at the current function's scope. It follows a **scope chain** — it looks outward through each containing scope until it either finds the variable or reaches the global scope and throws a `ReferenceError`.

```javascript
const level1 = 'global';

function outer() {
    const level2 = 'outer function';
  
    function middle() {
        const level3 = 'middle function';
      
        function inner() {
            const level4 = 'inner function';
          
            // The engine searches outward through each scope
            console.log(level4); // found in own scope — done
            console.log(level3); // not here, look up — found in middle
            console.log(level2); // not here, look up — found in outer
            console.log(level1); // not here, look up — found in global
        }
      
        inner();
    }
  
    middle();
}

outer();
```

The scope chain only goes **outward** — never inward, never sideways. An outer function cannot see inside its inner functions. Two sibling functions cannot see each other's variables.

```javascript
function sibling1() {
    const secret = 'only mine';
}

function sibling2() {
    console.log(secret); // ReferenceError — can't see into sibling1
}
```

This outward-only traversal is a direct consequence of lexical scoping — it's determined by where functions are  *written* , not how they relate at runtime.

## Now — What Exactly Is a Closure?

Here is the precise definition:

**A closure is created when a function is defined inside another function, and the inner function references variables from the outer function's scope. The inner function retains a live reference to those variables even after the outer function has returned and its execution context has been removed from the call stack.**

Let's dissect this with the simplest possible example:

```javascript
function outer() {
    const count = 0; // this variable lives in outer's scope
  
    function inner() {
        // inner REFERENCES count from outer's scope
        // This reference is what creates the closure
        console.log(count);
    }
  
    return inner; // we return the inner function itself
}

const myFunc = outer(); // outer runs, returns inner, outer's frame pops off the stack
myFunc(); // 0 — but wait... how does inner still know about count?
           // outer finished executing! its stack frame is gone!
```

This is the magic. When `outer()` finishes, its execution context is removed from the call stack. Normally, all variables in that context (`count`) would be eligible for garbage collection. But because `inner` was returned and is still referenced by `myFunc`, and because `inner` has a reference to `count`, the garbage collector sees that `count` is still reachable. It cannot be collected.

The `inner` function has "closed over" `count` — it has captured that variable and carries it with it wherever it goes.

## The Critical Detail — Closures Capture References, Not Values

This is the single most important thing to understand about closures, and the source of the most common closure-related bugs.

```javascript
function makeCounter() {
    let count = 0; // one single 'count' variable in memory
  
    return {
        increment() { count++; },
        decrement() { count--; },
        getCount()  { return count; }
    };
}

const counter = makeCounter();

counter.increment();
counter.increment();
counter.increment();
counter.decrement();
console.log(counter.getCount()); // 2
```

All three methods — `increment`, `decrement`, and `getCount` — close over the **same** `count` variable. Not copies of it — the actual variable in memory. When `increment` does `count++`, it's modifying the real `count`. When `getCount` reads it, it reads the same real `count`. They share the same closed-over variable.

This is what makes closures so powerful — they don't freeze a snapshot of a value, they maintain a live, persistent connection to a variable that outlives the function that created it.

### The Classic Loop Bug — Caused by Exactly This

This is one of the most famous JavaScript interview questions:

```javascript
// The BROKEN version
for (var i = 0; i < 3; i++) {
    setTimeout(function() {
        console.log(i); // Expected: 0, 1, 2
    }, 1000);           // Actual: 3, 3, 3
}
```

Why does this print `3, 3, 3`? Because `var` is function-scoped, there is only **one** `i` variable shared across all three iterations. All three callbacks close over that same single `i`. By the time the `setTimeout` callbacks fire (1000ms later), the loop has finished and `i` is `3`. All three closures read the same `i`, which is now `3`.

**Fix 1 — Use `let` (block scoped):**

```javascript
for (let i = 0; i < 3; i++) {
    setTimeout(function() {
        console.log(i); // 0, 1, 2 — correct!
    }, 1000);
}
```

`let` creates a **new binding** of `i` for each iteration of the loop. Each callback closes over its own separate `i`. When the callbacks fire, each one reads its own unique `i`.

**Fix 2 — Use an IIFE to create a new scope (the pre-ES6 solution):**

```javascript
for (var i = 0; i < 3; i++) {
    (function(capturedI) {
        setTimeout(function() {
            console.log(capturedI); // 0, 1, 2 — correct!
        }, 1000);
    })(i); // immediately pass current i as argument
}
```

The IIFE creates a new function scope for each iteration. `capturedI` is a new variable in each IIFE invocation, receiving the current value of `i`. Each callback closes over its own `capturedI`.

## The Closure Scope Chain — Visualizing What's in Memory

When a closure is created, the engine builds what's called a **Closure Environment Record** — a package containing all the variables the inner function references from outer scopes. Let's visualize this:

```javascript
function createMultiplier(multiplier) {
    // 'multiplier' is captured in the closure environment
  
    return function(number) {
        // 'number' is in this function's own scope
        // 'multiplier' is accessed from the closure environment
        return number * multiplier;
    };
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
const quadruple = createMultiplier(4);

// Each function has its OWN closure environment
// double's closure:    { multiplier: 2 }
// triple's closure:    { multiplier: 3 }
// quadruple's closure: { multiplier: 4 }

double(5);    // 10
triple(5);    // 15
quadruple(5); // 20
```

Each call to `createMultiplier` creates a completely separate closure environment. `double`, `triple`, and `quadruple` are three different function objects, each carrying their own `multiplier` value in their own private closure. They don't interfere with each other at all.

## Practical Closure Patterns

Now let's look at how closures are used to solve real problems. These are patterns you'll see and use constantly.

### Pattern 1 — Data Privacy and Encapsulation (The Module Pattern)

Before ES modules existed, closures were how JavaScript developers created private state — data that couldn't be accessed or modified from outside:

```javascript
function createBankAccount(initialBalance) {
    // 'balance' is completely private — no direct access from outside
    let balance = initialBalance;
    const transactionHistory = [];
  
    return {
        deposit(amount) {
            if (amount <= 0) throw new Error('Amount must be positive');
            balance += amount;
            transactionHistory.push({ type: 'deposit', amount });
            return balance;
        },
      
        withdraw(amount) {
            if (amount > balance) throw new Error('Insufficient funds');
            balance -= amount;
            transactionHistory.push({ type: 'withdrawal', amount });
            return balance;
        },
      
        getBalance() {
            return balance; // read-only access to balance
        },
      
        getHistory() {
            // Return a copy so the original can't be mutated from outside
            return [...transactionHistory];
        }
    };
}

const account = createBankAccount(1000);

account.deposit(500);    // 1500
account.withdraw(200);   // 1300
account.getBalance();    // 1300

// Try to access balance directly
console.log(account.balance); // undefined — completely private!

// Try to modify it directly
account.balance = 0; // does nothing to the real balance
account.getBalance(); // still 1300
```

This is genuine encapsulation — not just a convention, but enforced by the language through closures. The only way to interact with `balance` is through the methods provided.

### Pattern 2 — Function Factories

A function factory is a function that creates and returns specialized functions based on arguments:

```javascript
function makeValidator(minLength, maxLength, allowedChars) {
    // All three params are captured in each validator's closure
    const regex = new RegExp(`^[${allowedChars}]+$`);
  
    return function validate(input) {
        if (input.length < minLength) {
            return { valid: false, error: `Minimum ${minLength} characters` };
        }
        if (input.length > maxLength) {
            return { valid: false, error: `Maximum ${maxLength} characters` };
        }
        if (!regex.test(input)) {
            return { valid: false, error: 'Invalid characters' };
        }
        return { valid: true };
    };
}

// Create specialized validators, each with their own closure
const validateUsername = makeValidator(3, 20, 'a-zA-Z0-9_');
const validatePin = makeValidator(4, 4, '0-9');
const validateName = makeValidator(2, 50, 'a-zA-Z ');

validateUsername('peter_parker'); // { valid: true }
validateUsername('x');            // { valid: false, error: 'Minimum 3 characters' }
validatePin('1234');              // { valid: true }
validatePin('12345');             // { valid: false, error: 'Maximum 4 characters' }
```

### Pattern 3 — Memoization (Caching Results)

This is directly from your PDF. The cache lives in the closure:

```javascript
function memoize(fn) {
    const cache = {}; // this lives in the closure — persists between calls
  
    return function(...args) {
        const key = JSON.stringify(args);
      
        if (key in cache) {
            console.log('Cache hit!');
            return cache[key];
        }
      
        console.log('Computing...');
        const result = fn.apply(this, args);
        cache[key] = result;
        return result;
    };
}

function expensiveCalculation(n) {
    // Simulate heavy computation
    let result = 0;
    for (let i = 0; i < n * 1000000; i++) result += i;
    return result;
}

const memoizedCalc = memoize(expensiveCalculation);

memoizedCalc(10); // Computing... (takes time)
memoizedCalc(10); // Cache hit! (instant)
memoizedCalc(10); // Cache hit! (instant)
memoizedCalc(20); // Computing... (takes time)
memoizedCalc(20); // Cache hit! (instant)
```

The `cache` object persists between calls because it's in the closure. Every call to `memoizedCalc` accesses the same `cache` — it accumulates results over time.

### Pattern 4 — Partial Application and Currying

Both of these patterns from your PDF are closure-based:

```javascript
// Partial application — fix some arguments now, provide rest later
function partial(fn, ...presetArgs) {
    return function(...laterArgs) {
        // presetArgs is captured in the closure
        return fn(...presetArgs, ...laterArgs);
    };
}

function sendRequest(method, baseURL, endpoint, data) {
    console.log(`${method} ${baseURL}${endpoint}`, data);
}

// Fix the first two arguments — they'll be in the closure
const postToAPI = partial(sendRequest, 'POST', 'https://api.example.com');

// Now we only need to provide the remaining args
postToAPI('/users', { name: 'Peter' });
// POST https://api.example.com/users { name: 'Peter' }

postToAPI('/orders', { item: 'laptop' });
// POST https://api.example.com/orders { item: 'laptop' }
```

### Pattern 5 — Debounce and Throttle (From Your PDF)

Now that you understand closures, let's re-examine debounce. The `timerID` variable is what makes it work — and it works because of closures:

```javascript
function debounce(fn, delay) {
    let timerID; // this lives in the closure — shared across all calls
  
    return function(...args) {
        // Every call to this returned function accesses the SAME timerID
        clearTimeout(timerID); // cancel the previous timer
      
        // Set a new timer — stored back into the SAME timerID
        timerID = setTimeout(() => {
            fn(...args);
        }, delay);
    };
}

const searchAPI = (query) => console.log(`Searching: ${query}`);
const debouncedSearch = debounce(searchAPI, 300);

// User types rapidly — only the last call fires after 300ms of silence
debouncedSearch('j');
debouncedSearch('ja');
debouncedSearch('jav');
debouncedSearch('java');
debouncedSearch('javas');
// ... 300ms passes ...
// Searching: javas  ← only this fires
```

Without the closure keeping `timerID` alive between calls, each call to `debouncedSearch` would have its own fresh `timerID` — and `clearTimeout` would never be able to cancel the previous call's timer. The entire debounce mechanism depends on the single shared `timerID` in the closure.

## Closures and the Event Loop — How They Interact

Closures interact in a particularly interesting way with async code. The closure environment persists as long as any function referencing it is alive — including callbacks waiting in the event queue:

```javascript
function setupRequest() {
    const requestData = { userId: 123, timestamp: Date.now() };
    const logs = [];
  
    fetch('/api/data')
        .then(response => {
            // This callback is a closure over requestData and logs
            // It might execute milliseconds or seconds from now
            // But requestData and logs are STILL alive because this
            // closure references them
            logs.push(`Response received`);
            return response.json();
        })
        .then(data => {
            // This is ALSO a closure over the same requestData and logs
            logs.push(`Data processed for user ${requestData.userId}`);
            console.log(logs);
        });
  
    // setupRequest returns here — its stack frame is gone
    // But requestData and logs persist because the closures above
    // still reference them and they're sitting in the microtask queue
}
```

This is a profound point: closures are what allow async callbacks to "remember" the context they were created in, even though the function that created them finished executing long ago and has been removed from the call stack.

## Closures in React Hooks

React hooks are entirely built on closures. Understanding this explains many common React bugs:

```javascript
function Counter() {
    const [count, setCount] = React.useState(0);
  
    React.useEffect(() => {
        const timer = setInterval(() => {
            // This callback closes over 'count'
            // But 'count' here is the value from when this effect ran
            // This is the "stale closure" problem
            console.log(count); // always logs 0!
            setCount(count + 1); // always sets to 0 + 1 = 1!
        }, 1000);
      
        return () => clearInterval(timer);
    }, []); // empty dependency array — effect runs once, closes over count=0
}
```

This "stale closure" bug is one of the most common React issues. The `setInterval` callback closes over `count` at the time the effect ran (when `count` was `0`). Even as `count` changes through re-renders, the closure still holds the old reference to the `count` from the original render.

The fix involves either using the functional form of setState (which doesn't rely on closed-over state) or adding `count` to the dependency array. Understanding closures is what makes this bug both comprehensible and fixable.

## Memory Implications of Closures

Connecting back to our memory discussion — closures keep variables alive in the heap. This is powerful but requires care:

```javascript
function heavyOperation() {
    // This large array takes significant memory
    const massiveArray = new Array(1000000).fill('data');
  
    // processedResult is a small summary
    const processedResult = massiveArray.length;
  
    return function getResult() {
        // This closure captures 'massiveArray' AND 'processedResult'
        // Even though we only USE processedResult,
        // massiveArray stays in memory because the closure
        // keeps the entire scope alive
        return processedResult;
    };
}

const getResult = heavyOperation();
// massiveArray is STILL in memory, even though we only need processedResult!
// The closure kept the whole scope alive
```

Modern JS engines are getting smarter about only capturing variables that are actually referenced, but this optimization isn't guaranteed. The safe approach is to be deliberate:

```javascript
function heavyOperation() {
    const massiveArray = new Array(1000000).fill('data');
    const processedResult = massiveArray.length;
  
    // massiveArray is no longer referenced after this point
    // The returned closure only captures processedResult
    // massiveArray becomes eligible for GC
  
    return function getResult() {
        return processedResult; // only this is in the closure
    };
}
```

## The Complete Mental Model

Here's the definitive mental model that unifies everything:

When a function is  **defined** , the engine attaches to it a reference to the **scope environment** where it was created. This is called the function's `[[Environment]]` internal slot — it's part of every function object, invisible to you but always there.

When the function is  **called** , a new execution context is created. This context has access to its own local variables AND to the outer scope environment attached to the function via `[[Environment]]`. This is what forms the scope chain.

When the function  **returns** , its execution context is removed from the call stack. But the scope environment itself stays in memory for as long as any function references it. If you returned an inner function that closes over that scope, the scope lives on — attached to the inner function via its own `[[Environment]]`.

This is the complete picture:

```
Function Object
├── Code (what to execute)
├── [[Environment]] ──────────────────→ Outer Scope Environment
│                                        ├── variable: value
│                                        ├── variable: value
│                                        └── [[OuterEnv]] ──→ Even Outer Scope
└── Other internal slots
```

Every time you create a closure, you're creating a function object that holds a reference to an environment record in the heap. That environment record holds the closed-over variables. The chain of environment records going outward is the scope chain.

Closures are not a special feature that you opt into — they are simply the natural consequence of functions being first-class values in a lexically scoped language. Any time a function references a variable from an outer scope and that function outlives the outer function's execution, you have a closure. Whether that's a callback, a returned function, an event handler, or a Promise `.then()` — the same mechanism is at work every single time.

This is why closures are everywhere in JavaScript. They're not a pattern on top of the language — they're baked into how the language fundamentally works.
