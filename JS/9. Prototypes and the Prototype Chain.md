# Prototypes and the Prototype Chain — The Complete Guide

---

## Start From the Fundamental Question

Before diving into syntax and code, let's ask the question that makes prototypes  *necessary* :

If you create 1000 `Person` objects, and every `Person` has a `greet()` method — do you really need 1000 separate copies of the same `greet` function sitting in memory? That's wasteful. Wouldn't it be better to have **one** `greet` function somewhere, and have all 1000 objects *share* it?

That shared location is the  **prototype** . Prototypes are JavaScript's solution to shared behavior and inheritance. Understanding this motivation makes everything else click naturally.

---

## What Is a Prototype?

In JavaScript,  **every object has a hidden internal property called `[[Prototype]]`** . This is not a property you see when you `console.log` an object — it's internal to the engine. It either points to another object (that object's prototype) or is `null` (meaning the chain ends here).

When you try to access a property on an object and it doesn't exist directly on that object, the engine doesn't just give up. It follows the `[[Prototype]]` link to the next object and looks there. If it's not there either, it follows that object's `[[Prototype]]` link. This continues until either the property is found or `[[Prototype]]` is `null`. This chain of linked objects is the  **Prototype Chain** .

This single mechanism is how JavaScript implements inheritance, shared methods, and all built-in functionality like `.map()`, `.filter()`, `.toString()`, and every other method you've ever called on arrays, objects, and strings.

---

## The `[[Prototype]]` in Plain Terms

Let's build this understanding from scratch with the simplest possible example:

```javascript
const animal = {
    breathe() {
        console.log('breathing...');
    }
};

const dog = {
    bark() {
        console.log('woof!');
    }
};

// Manually set animal as the prototype of dog
// (we'll see cleaner ways to do this shortly)
Object.setPrototypeOf(dog, animal);

dog.bark();    // 'woof!' — found directly on dog
dog.breathe(); // 'breathing...' — NOT on dog, found on animal via prototype
```

When you call `dog.breathe()`, the engine does this:

1. Look for `breathe` on `dog` directly — not found
2. Follow `dog`'s `[[Prototype]]` link to `animal`
3. Look for `breathe` on `animal` — found! Execute it.

This lookup is called **property shadowing** — if the property IS found on the object itself, it "shadows" (overrides) any property with the same name further up the chain.

```javascript
const animal = {
    sound() {
        console.log('some generic sound');
    }
};

const dog = {
    sound() {
        console.log('woof!'); // shadows animal.sound
    }
};

Object.setPrototypeOf(dog, animal);

dog.sound(); // 'woof!' — own property found first, chain not traversed
```

## How JavaScript Actually Stores Prototypes — `prototype` vs `__proto__`

This is where most tutorials lose people, because there are **two different things** with confusingly similar names. Let's clarify them completely.

### `__proto__` — The Accessor for `[[Prototype]]`

`__proto__` (pronounced "dunder proto") is a getter/setter that lets you read and write the internal `[[Prototype]]` of any object. It's the visible interface to the hidden `[[Prototype]]` slot.

```javascript
const peter = { name: 'Peter' };
const human = { species: 'Homo sapiens' };

peter.__proto__ = human; // set human as peter's prototype

console.log(peter.name);    // 'Peter' — own property
console.log(peter.species); // 'Homo sapiens' — from prototype chain
```

Every object you create has `__proto__`. It's the link in the chain.

### `prototype` — The Property on Constructor Functions

`prototype` (no underscores) is a completely different thing. It is a **regular property** that exists on **functions** (specifically constructor functions and classes). When you use `new` to create an object from a constructor function, the new object's `[[Prototype]]` (`__proto__`) is set to the constructor function's `prototype` property.

```javascript
function Person(name) {
    this.name = name;
}

// Person.prototype is an object that exists right now,
// even before we create any Person instances
console.log(Person.prototype);
// { constructor: [Function: Person] }

// When we use 'new', the new object's __proto__ is set
// to Person.prototype
const peter = new Person('Peter');

console.log(peter.__proto__ === Person.prototype); // true — same object!
console.log(peter.__proto__ === Person.prototype); // true
```

This is the crucial relationship:

```
peter.__proto__ ────────────────→ Person.prototype
                                  {
                                    constructor: Person,
                                    greet: function...  ← shared by ALL instances
                                  }
```

Every instance created with `new Person()` shares the same `Person.prototype` object. Methods added to `Person.prototype` are available to all instances through the prototype chain — without being copied onto each instance.

```javascript
function Person(name) {
    this.name = name; // own property — each instance gets its own copy
}

// Add method to prototype — shared by ALL instances
Person.prototype.greet = function() {
    console.log(`Hello, I'm ${this.name}`);
};

Person.prototype.species = 'Human'; // shared property

const peter = new Person('Peter');
const tony  = new Person('Tony');

peter.greet(); // "Hello, I'm Peter"
tony.greet();  // "Hello, I'm Tony"

// Both instances share the SAME greet function
console.log(peter.greet === tony.greet); // true — exact same function in memory!

// Both instances share the SAME species property
console.log(peter.species); // 'Human'
console.log(tony.species);  // 'Human'

// Own properties are separate per instance
console.log(peter.name === tony.name); // false — different values
```

This is the efficiency win — one `greet` function in memory, shared by 1000 instances.

## The Complete Prototype Chain Visualization

Let's trace the FULL prototype chain for a typical object:

```javascript
function Person(name) {
    this.name = name;
}

Person.prototype.greet = function() {
    return `Hello, I'm ${this.name}`;
};

const peter = new Person('Peter');
```

The chain looks like this:

```
peter
├── name: 'Peter'                    (own property)
└── [[Prototype]] ──────────────────→ Person.prototype
                                      ├── greet: function
                                      ├── constructor: Person
                                      └── [[Prototype]] ──→ Object.prototype
                                                            ├── toString: function
                                                            ├── hasOwnProperty: function
                                                            ├── valueOf: function
                                                            └── [[Prototype]] ──→ null
```

When you call `peter.greet()`:

1. Look on `peter` — not found
2. Look on `Person.prototype` — found! Execute.

When you call `peter.toString()`:

1. Look on `peter` — not found
2. Look on `Person.prototype` — not found
3. Look on `Object.prototype` — found! Execute.

When you access `peter.nonExistent`:

1. Look on `peter` — not found
2. Look on `Person.prototype` — not found
3. Look on `Object.prototype` — not found
4. `[[Prototype]]` is `null` — chain ends
5. Return `undefined`

This is why every JavaScript object has access to methods like `toString()`, `hasOwnProperty()`, `valueOf()` — they all live on `Object.prototype` at the top of every object's prototype chain.

---

## `Object.prototype` — The Top of Every Chain

`Object.prototype` is the ancestor of almost every object in JavaScript. It's where the built-in methods that every object can use are defined:

```javascript
const obj = { name: 'Peter' };

// These all come from Object.prototype
obj.toString();           // "[object Object]"
obj.hasOwnProperty('name'); // true
obj.valueOf();            // { name: 'Peter' }

// Verify they come from Object.prototype
console.log(obj.toString === Object.prototype.toString); // true
```

`Object.prototype` itself has `[[Prototype]]` of `null` — this is where the chain always ends.

## Prototypal Inheritance — The Full Concept

Classical inheritance (like in Java or C++) creates a rigid hierarchy of classes where a subclass *copies* the behavior of its parent. JavaScript's prototypal inheritance is different — it's based on  **delegation** . Objects don't copy behavior from their prototype; they **delegate** property lookups to it.

When `peter.greet()` is called and `greet` isn't on `peter`, `peter` *delegates* that lookup to its prototype. The prototype handles the call. `peter` never owned `greet` — it just knows where to ask.

This distinction matters:

* Classical: child gets a copy of parent's methods at construction time
* Prototypal: child delegates to parent at lookup time — if parent changes, child sees the change immediately

```javascript
function Animal(name) {
    this.name = name;
}

Animal.prototype.describe = function() {
    return `I am ${this.name}`;
};

const cat = new Animal('Cat');

console.log(cat.describe()); // 'I am Cat'

// Change the prototype AFTER creating the instance
// Prototypal delegation means cat sees this immediately
Animal.prototype.describe = function() {
    return `I am ${this.name} and I am an animal`;
};

console.log(cat.describe()); // 'I am Cat and I am an animal'
// cat didn't change — its PROTOTYPE changed, and cat delegates to it
```

## Creating Prototype Chains — Implementing Inheritance

The whole point of prototype chains is to implement inheritance — where one type of object inherits behavior from another. Let's build this step by step.

### Step 1 — The Parent Constructor

```javascript
function Animal(name, sound) {
    this.name = name;
    this.sound = sound;
}

Animal.prototype.speak = function() {
    return `${this.name} says ${this.sound}`;
};

Animal.prototype.describe = function() {
    return `I am ${this.name}`;
};
```

### Step 2 — The Child Constructor

```javascript
function Dog(name) {
    // Call parent constructor to set up own properties
    // 'this' is the new Dog instance
    Animal.call(this, name, 'woof');
  
    // Dog-specific own properties
    this.tricks = [];
}
```

### Step 3 — Link the Prototype Chain

This is the critical step. We need `Dog.prototype` to have `Animal.prototype` in its chain:

```javascript
// Create a new object whose [[Prototype]] is Animal.prototype
// and set it as Dog.prototype
Dog.prototype = Object.create(Animal.prototype);

// IMPORTANT: restore the constructor reference
// Object.create wiped out the original Dog.prototype which had constructor: Dog
Dog.prototype.constructor = Dog;
```

### Step 4 — Add Dog-Specific Methods

```javascript
Dog.prototype.learn = function(trick) {
    this.tricks.push(trick);
    return this;
};

Dog.prototype.perform = function() {
    if (this.tricks.length === 0) {
        return `${this.name} doesn't know any tricks`;
    }
    return `${this.name} performs: ${this.tricks.join(', ')}`;
};
```

### Step 5 — Use It

```javascript
const rex = new Dog('Rex');

rex.speak();    // 'Rex says woof' — from Animal.prototype
rex.describe(); // 'I am Rex' — from Animal.prototype
rex.learn('sit').learn('shake').learn('roll over');
rex.perform();  // 'Rex performs: sit, shake, roll over'

// Verify the chain
console.log(rex instanceof Dog);    // true
console.log(rex instanceof Animal); // true — because chain includes Animal.prototype
```

The prototype chain for `rex` now looks like:

```
rex
├── name: 'Rex'
├── sound: 'woof'
├── tricks: ['sit', 'shake', 'roll over']
└── [[Prototype]] ──────→ Dog.prototype
                          ├── constructor: Dog
                          ├── learn: function
                          ├── perform: function
                          └── [[Prototype]] ──→ Animal.prototype
                                                ├── speak: function
                                                ├── describe: function
                                                └── [[Prototype]] ──→ Object.prototype
                                                                      └── [[Prototype]] → null
```

## The `__proto__` Warning

`__proto__` works, but it comes with important caveats that you need to know.

**It is deprecated as a direct assignment mechanism.** The double-underscore naming convention signals that it's an internal/legacy property. It was never part of the official ECMAScript specification initially — it was a browser extension that became so widely used that it was eventually standardized in ES6, but as a legacy feature.

**It is slow.** Changing the `[[Prototype]]` of an existing object with `__proto__` is a very slow operation in all JavaScript engines. When you change a prototype, the engine has to rebuild all its internal optimizations for that object and every object that references it. V8 specifically warns about this in its optimization guides.

**Use the official alternatives instead:**

```javascript
// READING the prototype — these are all equivalent
Object.getPrototypeOf(obj);  // ✅ Official, recommended
obj.__proto__;                // ⚠️  Works but deprecated

// SETTING the prototype at creation — preferred
const child = Object.create(parent); // ✅ Create new object with parent as prototype

// SETTING the prototype after creation — avoid if possible
Object.setPrototypeOf(child, parent); // ⚠️  Official but slow
child.__proto__ = parent;             // ❌  Deprecated and slow
```

The golden rule:  **set prototypes at object creation time, not after** . `Object.create()` is the right tool for this.

## `Object.create()` — The Clean Way

`Object.create(proto)` creates a brand new object whose `[[Prototype]]` is set to `proto`. It's the cleanest, most explicit way to set up prototype relationships.

```javascript
const personPrototype = {
    greet() {
        return `Hello, I'm ${this.name}`;
    },
  
    introduce() {
        return `I'm ${this.name} and I'm ${this.age} years old`;
    }
};

// Create an object with personPrototype as its prototype
const peter = Object.create(personPrototype);
peter.name = 'Peter';
peter.age = 25;

peter.greet();     // "Hello, I'm Peter"
peter.introduce(); // "I'm Peter and I'm 25 years old"

console.log(Object.getPrototypeOf(peter) === personPrototype); // true
```

### `Object.create(null)` — Creating Truly Empty Objects

Passing `null` creates an object with NO prototype chain — not even `Object.prototype`. This creates a truly empty object, useful for dictionaries/maps where you don't want prototype pollution:

```javascript
// Regular object — has Object.prototype methods
const regularObj = {};
console.log(regularObj.toString);     // function — from Object.prototype
console.log(regularObj.hasOwnProperty); // function — from Object.prototype

// Truly empty object — no prototype chain at all
const pureDict = Object.create(null);
console.log(pureDict.toString);       // undefined — nothing inherited
console.log(pureDict.hasOwnProperty); // undefined

// Safe to use as a pure key-value store
pureDict['constructor'] = 'safe';    // no conflict with Object.prototype
pureDict['toString'] = 'also safe';  // no conflict
```

### `Object.create()` with Property Descriptors

`Object.create()` also accepts a second argument — property descriptors for the new object:

```javascript
const peter = Object.create(personPrototype, {
    name: {
        value: 'Peter Parker',
        writable: true,
        enumerable: true,
        configurable: true
    },
    age: {
        value: 25,
        writable: true,
        enumerable: true,
        configurable: true
    }
});

peter.greet(); // "Hello, I'm Peter Parker"
```

### Using `Object.create()` for Clean Inheritance

This is the proper pattern for setting up inheritance:

```javascript
function Animal(name) {
    this.name = name;
}

Animal.prototype.speak = function() {
    return `${this.name} makes a sound`;
};

function Dog(name, breed) {
    Animal.call(this, name); // inherit own properties
    this.breed = breed;
}

// The RIGHT way — Object.create instead of new Animal()
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;

Dog.prototype.bark = function() {
    return `${this.name} barks!`;
};

const rex = new Dog('Rex', 'Labrador');
rex.speak(); // 'Rex makes a sound'
rex.bark();  // 'Rex barks!'
```

Why `Object.create(Animal.prototype)` and not `new Animal()`?

Because `new Animal()` would run the `Animal` constructor function, which expects arguments and might have side effects. We don't want to construct an actual Animal — we just want an object that inherits from `Animal.prototype`. `Object.create` gives us exactly that, cleanly.

## ES6 Classes — Prototype Under the Hood

Modern JavaScript gives us the `class` syntax, which is cleaner but is 100% prototype-based under the hood. Classes are **syntactic sugar** over the prototype system:

```javascript
class Animal {
    constructor(name) {
        this.name = name; // own property on each instance
    }
  
    // Added to Animal.prototype automatically
    speak() {
        return `${this.name} makes a sound`;
    }
  
    // Static method — on Animal itself, not Animal.prototype
    static create(name) {
        return new Animal(name);
    }
}

class Dog extends Animal {
    constructor(name, breed) {
        super(name); // calls Animal constructor — sets this.name
        this.breed = breed;
    }
  
    // Added to Dog.prototype
    bark() {
        return `${this.name} barks!`;
    }
  
    // Overrides Animal.prototype.speak — property shadowing
    speak() {
        return `${this.name} barks: woof!`;
    }
}

const rex = new Dog('Rex', 'Labrador');
rex.speak(); // 'Rex barks: woof!' — Dog.prototype.speak shadows Animal.prototype.speak
rex.bark();  // 'Rex barks!'

// Under the hood — it's still just prototypes
console.log(Object.getPrototypeOf(rex) === Dog.prototype);            // true
console.log(Object.getPrototypeOf(Dog.prototype) === Animal.prototype); // true
```

The `extends` keyword sets up the prototype chain — `Dog.prototype`'s `[[Prototype]]` is set to `Animal.prototype`. The `super()` call in the constructor is equivalent to `Animal.call(this, name)`.

## Extending Built-in Objects

You can extend built-in prototypes like `Array.prototype`, `String.prototype`, `Date.prototype` — but this comes with serious caveats we'll cover right after.

### Date Object Extension

```javascript
// Adding a method to format dates
Date.prototype.formatDate = function() {
    const day   = String(this.getDate()).padStart(2, '0');
    const month = String(this.getMonth() + 1).padStart(2, '0'); // months are 0-indexed
    const year  = this.getFullYear();
    return `${day}/${month}/${year}`;
};

Date.prototype.addDays = function(days) {
    const result = new Date(this);
    result.setDate(result.getDate() + days);
    return result;
};

Date.prototype.isWeekend = function() {
    const day = this.getDay(); // 0 = Sunday, 6 = Saturday
    return day === 0 || day === 6;
};

// Now ALL Date instances have these methods
const today = new Date();
console.log(today.formatDate()); // '08/06/2026'

const nextWeek = today.addDays(7);
console.log(nextWeek.formatDate()); // '15/06/2026'

console.log(today.isWeekend()); // depends on the day
```

### Array Method Modification — Deep Dive

You can override or add methods to `Array.prototype`:

```javascript
// Adding a useful method that doesn't exist natively
Array.prototype.sum = function() {
    return this.reduce((total, n) => total + n, 0);
};

Array.prototype.average = function() {
    if (this.length === 0) return 0;
    return this.sum() / this.length;
};

Array.prototype.unique = function() {
    return [...new Set(this)];
};

Array.prototype.groupBy = function(keyFn) {
    return this.reduce((groups, item) => {
        const key = keyFn(item);
        groups[key] = groups[key] || [];
        groups[key].push(item);
        return groups;
    }, {});
};

// Usage
[1, 2, 3, 4, 5].sum();     // 15
[1, 2, 3, 4, 5].average(); // 3
[1, 2, 2, 3, 3, 3].unique(); // [1, 2, 3]

const people = [
    { name: 'Peter', dept: 'Engineering' },
    { name: 'Tony',  dept: 'Engineering' },
    { name: 'Steve', dept: 'Management' }
];

people.groupBy(p => p.dept);
// {
//   Engineering: [{ name: 'Peter'... }, { name: 'Tony'... }],
//   Management:  [{ name: 'Steve'... }]
// }
```

### Overriding Existing Built-in Methods

You can also override existing methods — but be extremely careful:

```javascript
// Override Array.prototype.map to log when it's used
const originalMap = Array.prototype.map;

Array.prototype.map = function(callback, thisArg) {
    console.log(`map called on array of length ${this.length}`);
    return originalMap.call(this, callback, thisArg);
};

[1, 2, 3].map(x => x * 2);
// logs: "map called on array of length 3"
// returns: [2, 4, 6]
```

## The Danger of Extending Built-in Prototypes — Monkey Patching

While extending built-in prototypes is possible, **it is generally considered bad practice** in application code. Here's exactly why:

**1. Name Collision Risk**

If a future version of JavaScript adds a native method with the same name as your custom one, your code breaks:

```javascript
// You add this today
Array.prototype.flatten = function() { /* your implementation */ };

// Tomorrow, ES2019 ships Array.prototype.flat() and flatten()
// Now your implementation conflicts with the native one
// Other libraries that call flatten() get YOUR version, not the native one
```

This actually happened — libraries that had added `Array.prototype.flatten` broke when ES2019 added `Array.prototype.flat` (which was originally going to be named `flatten`). It was called "SmooshGate".

**2. Pollutes Every Array in the Application**

Your addition appears in `for...in` loops unless you use `Object.defineProperty` with `enumerable: false`:

```javascript
Array.prototype.sum = function() {
    return this.reduce((t, n) => t + n, 0);
};

const arr = [1, 2, 3];

// This is a problem
for (let key in arr) {
    console.log(key); // '0', '1', '2', 'sum' — sum appears!
}
```

Fix with non-enumerable definition:

```javascript
Object.defineProperty(Array.prototype, 'sum', {
    value: function() {
        return this.reduce((t, n) => t + n, 0);
    },
    writable: true,
    enumerable: false,   // won't show in for...in
    configurable: true
});

const arr = [1, 2, 3];
for (let key in arr) {
    console.log(key); // '0', '1', '2' — 'sum' doesn't appear
}
arr.sum(); // 6 — still works
```

**3. Third-Party Library Conflicts**

If you and a library you use both extend `Array.prototype` with different implementations of the same method name, one will silently overwrite the other.

**The safe alternatives:**

```javascript
// Option 1 — Utility functions (preferred in modern codebases)
function arraySum(arr) {
    return arr.reduce((t, n) => t + n, 0);
}

// Option 2 — Extend your own class instead of the built-in
class EnhancedArray extends Array {
    sum() {
        return this.reduce((t, n) => t + n, 0);
    }
  
    average() {
        return this.sum() / this.length;
    }
}

const arr = new EnhancedArray(1, 2, 3, 4, 5);
arr.sum();     // 15
arr.average(); // 3
arr.map(x => x * 2); // [2, 4, 6, 8, 10] — inherited from Array
```

Extending your own subclass gives you all the added methods without touching the built-in prototype that everything else depends on.

## Checking the Prototype Chain — Useful Methods

```javascript
// instanceof — checks if prototype is anywhere in the chain
rex instanceof Dog;    // true
rex instanceof Animal; // true
rex instanceof Object; // true — everything is

// isPrototypeOf — check if object is in another's chain
Dog.prototype.isPrototypeOf(rex);    // true
Animal.prototype.isPrototypeOf(rex); // true

// getPrototypeOf — get the direct prototype
Object.getPrototypeOf(rex) === Dog.prototype; // true

// hasOwnProperty — check if property is OWN vs inherited
rex.hasOwnProperty('name');  // true — own property
rex.hasOwnProperty('speak'); // false — inherited from Animal.prototype

// Object.keys — only own enumerable properties
Object.keys(rex); // ['name', 'breed'] — prototype methods not included

// for...in — own AND inherited enumerable properties
for (let key in rex) {
    console.log(key); // includes name, breed, AND speak, bark, etc.
  
    // Filter to own properties only
    if (rex.hasOwnProperty(key)) {
        console.log('own:', key);
    }
}
```

## The Complete Mental Model

Here is the unified picture of everything we covered:

Every object in JavaScript is a node in a linked list of objects — the prototype chain. Each node can have its own properties, and delegates any property lookups it can't satisfy to the next node in the chain.

Constructor functions (and classes) use their `prototype` property as the shared node that all their instances link to. This single shared node holds all the methods — one copy in memory, accessible to all instances through delegation.

`Object.create()` is the cleanest way to create prototype relationships explicitly. ES6 classes are syntactic sugar that set up the same prototype chain relationships automatically, with cleaner syntax.

The chain always ends at `Object.prototype`, whose own `[[Prototype]]` is `null`.

This entire system — every method call on every array, string, object, and class instance you've ever used — runs through this one consistent mechanism of prototype chain traversal. It's the single most pervasive feature in all of JavaScript.
